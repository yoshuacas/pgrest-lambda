# Design Documents

This directory contains design documents for features
and changes to the project. Each design doc should be
a single Markdown file named after the feature (e.g.,
`some-feature.md`).

Run `rring designs` to list all design documents.

## Structure

A design document should include the following
sections:

1. **Overview** — A concise summary of the feature.

2. **Current CX / Concepts** — Relevant existing
   behavior that the design builds on or changes.

3. **Proposed CX / CX Specification** — The
   user-facing experience: command syntax, validation
   rules, error messages, and example output.

4. **Technical Design** — Internal implementation:
   data model changes, new types, service methods,
   validation logic.

5. **Code Architecture / File Changes** — Which files
   are created or modified.

6. **Testing Strategy** — Unit and integration test
   coverage plan.

7. **Implementation Order** — Sequenced steps for
   incremental delivery.

8. **Open Questions** (optional) — Unresolved
   decisions or trade-offs.

# Task 01: End-to-End Tests for Resource Embedding

Agent: implementer
Design: docs/design/resource-embedding.md

## Objective

Write integration tests that exercise every resource embedding
behavior specified in the design document. All tests must compile
and all tests must fail (the feature does not exist yet).

## Test File

`test/integration/embedding.test.mjs`

This file follows the same pattern as the existing integration
tests in `test/integration/postgres.test.mjs`: it uses
`node:test`, `node:assert/strict`, the shared `makeEvent` helper,
and `createPgrest` from `src/index.mjs`.

## Test Schema

Create these tables with FK constraints for the test database.
Add this as `EMBEDDING_SCHEMA_SQL` in the test file (do not
modify `test/integration/helpers.mjs`):

```sql
DROP TABLE IF EXISTS order_items CASCADE;
DROP TABLE IF EXISTS products CASCADE;
DROP TABLE IF EXISTS orders CASCADE;
DROP TABLE IF EXISTS customers CASCADE;
DROP TABLE IF EXISTS addresses CASCADE;

CREATE TABLE customers (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY (CACHE 1)
    PRIMARY KEY,
  name TEXT NOT NULL,
  email TEXT
);

CREATE TABLE addresses (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY (CACHE 1)
    PRIMARY KEY,
  street TEXT NOT NULL,
  city TEXT NOT NULL
);

CREATE TABLE orders (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY (CACHE 1)
    PRIMARY KEY,
  customer_id BIGINT REFERENCES customers(id),
  billing_address_id BIGINT REFERENCES addresses(id),
  shipping_address_id BIGINT REFERENCES addresses(id),
  amount NUMERIC(10,2) NOT NULL
);

CREATE TABLE products (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY (CACHE 1)
    PRIMARY KEY,
  name TEXT NOT NULL,
  price NUMERIC(10,2) NOT NULL
);

CREATE TABLE order_items (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY (CACHE 1)
    PRIMARY KEY,
  order_id BIGINT REFERENCES orders(id),
  product_id BIGINT REFERENCES products(id),
  quantity INT NOT NULL DEFAULT 1
);
```

## Seed Data

```sql
INSERT INTO customers (id, name, email) VALUES
  (1, 'Alice', 'alice@test.com'),
  (2, 'Bob', 'bob@test.com');

INSERT INTO addresses (id, street, city) VALUES
  (1, '123 Billing St', 'New York'),
  (2, '456 Shipping Ave', 'Boston');

INSERT INTO orders (id, customer_id, billing_address_id,
    shipping_address_id, amount) VALUES
  (1, 1, 1, 2, 99.00),
  (2, 1, NULL, NULL, 50.00),
  (3, 2, 1, 1, 75.00);

INSERT INTO products (id, name, price) VALUES
  (1, 'Widget', 9.99),
  (2, 'Gadget', 19.99);

INSERT INTO order_items (id, order_id, product_id, quantity) VALUES
  (1, 1, 1, 2),
  (2, 1, 2, 1),
  (3, 3, 1, 5);
```

Note: Order 2 has `customer_id=1` (Alice) but NULL address FKs.
Customer Bob (id=2) has one order (id=3). This setup lets us
test null embeds and empty arrays.

## Test Cases

### Many-to-one embed

```
GET /rest/v1/orders?select=id,amount,customers(name,email)
```

- Assert HTTP 200
- Assert each order row has a `customers` key
- Assert order 1 has `customers: { name: 'Alice', email: 'alice@test.com' }`
- Assert `customers` is a plain object (not an array)

### Many-to-one embed returns null for NULL FK

```
GET /rest/v1/orders?select=id,customers(name)&customer_id=is.null
```

Use seed data: insert an order with `customer_id = NULL`.

- Assert order row has `customers: null`

Note: Order 2 in the seed has `customer_id=1` (not NULL). To
test null FK, either add a seed order with NULL customer_id or
use the existing `billing_address_id` NULL case on order 2:

```
GET /rest/v1/orders?select=id,addresses!billing_address_id(street)&id=eq.2
```

- Assert order 2 has billing address embed as `null` (since
  `billing_address_id` is NULL for order 2)

### One-to-many embed

```
GET /rest/v1/customers?select=id,orders(id,amount)
```

- Assert HTTP 200
- Assert customer 1 (Alice) has `orders` as an array with 2 items
- Assert customer 2 (Bob) has `orders` as an array with 1 item

### One-to-many embed returns empty array

Add a customer with no orders to the seed, or check: insert
a new customer and query:

```
POST /rest/v1/customers { name: 'Charlie', email: 'c@test.com' }
GET /rest/v1/customers?select=id,orders(id)&name=eq.Charlie
```

- Assert `orders: []` (empty array, not null)

### Wildcard parent and child

```
GET /rest/v1/orders?select=*,customers(*)
```

- Assert each order has all its own columns (`id`, `customer_id`,
  `billing_address_id`, `shipping_address_id`, `amount`)
- Assert `customers` key has all customer columns (`id`, `name`,
  `email`)

### Aliased embed

```
GET /rest/v1/orders?select=id,buyer:customers(name)
```

- Assert JSON key is `buyer`, not `customers`
- Assert `buyer` is an object with `name` property

### Nested embed (2 levels)

```
GET /rest/v1/orders?select=id,order_items(id,products(name))
```

- Assert order 1 has `order_items` array with 2 items
- Assert each item has `products` object with `name` key
- Assert product names include 'Widget' and 'Gadget'

### Inner join (one-to-many)

```
GET /rest/v1/customers?select=id,name,orders!inner(id)
```

- Assert only customers who have orders are returned
- All returned customers have non-empty `orders` array

Insert a customer with no orders before this test. That
customer should NOT appear in results.

### Inner join (many-to-one)

```
GET /rest/v1/orders?select=id,customers!inner(name)
```

- Assert only orders where `customer_id IS NOT NULL` are returned
- If any seed orders have NULL customer_id, they should be excluded

### Disambiguation with !hint

```
GET /rest/v1/orders?select=*,billing:addresses!billing_address_id(*),shipping:addresses!shipping_address_id(*)
```

- Assert each order has both `billing` and `shipping` keys
- Assert order 1 has `billing.street = '123 Billing St'`
- Assert order 1 has `shipping.street = '456 Shipping Ave'`
- Assert order 2 has `billing: null` and `shipping: null`
  (both address FKs are NULL on order 2)

### Error: no relationship found (PGRST200)

```
GET /rest/v1/orders?select=id,nonexistent(name)
```

- Assert HTTP 400
- Assert response body has `code: 'PGRST200'`
- Assert message contains 'orders' and 'nonexistent'

### Error: ambiguous relationship (PGRST201)

```
GET /rest/v1/orders?select=id,addresses(street)
```

(No !hint provided, but orders has two FKs to addresses)

- Assert HTTP 300
- Assert response body has `code: 'PGRST201'`
- Assert `details` is an array with 2 entries
- Assert `hint` field suggests using `!hint` syntax

### Error: column not found in embed (PGRST204)

```
GET /rest/v1/orders?select=id,customers(nonexistent_col)
```

- Assert HTTP 400
- Assert response body has `code: 'PGRST204'`
- Assert message references 'nonexistent_col' and 'customers'

### Space handling in select

```
GET /rest/v1/orders?select=id, customers(name, email)
```

(Spaces after commas, matching supabase-js output)

- Assert same result as `select=id,customers(name,email)`
  (spaces trimmed, no errors)

### Backward compatibility: flat select unchanged

```
GET /rest/v1/orders?select=id,amount
```

- Assert HTTP 200
- Assert rows have `id` and `amount` only (no extra keys)
- Assert this still works identically to pre-embedding behavior

### Backward compatibility: wildcard select unchanged

```
GET /rest/v1/orders?select=*
```

- Assert returns all columns of orders table
- No embedding keys present

### Backward compatibility: no select param

```
GET /rest/v1/orders
```

- Assert returns all columns (default `*` behavior)

### Filters work alongside embeds

```
GET /rest/v1/orders?select=id,customers(name)&amount=gt.50
```

- Assert only orders with amount > 50 are returned
- Assert each still has `customers` embed

### return=representation with embeds (POST)

```
POST /rest/v1/orders
  body: { customer_id: 2, amount: 200.00 }
  headers: { Prefer: 'return=representation' }
  query: { select: 'id,amount,customers(name)' }
```

- Assert HTTP 201
- Assert response includes the inserted row with embedded
  `customers` object (name: 'Bob')

### return=representation with embeds (PATCH)

```
PATCH /rest/v1/orders?id=eq.1
  body: { amount: 999.00 }
  headers: { Prefer: 'return=representation' }
  query: { select: 'id,amount,customers(name)' }
```

- Assert HTTP 200
- Assert response includes updated row with `amount: 999.00`
  and embedded `customers` object

### Authorization on embeds (silent exclusion)

This test requires Cedar policy setup that restricts access
to the `customers` table for a specific role/user while
allowing access to `orders`.

```
GET /rest/v1/orders?select=id,customers(name)
  (as authenticated user who can read orders but not customers)
```

- Assert HTTP 200 (not 403)
- Assert orders are returned
- Assert `customers` embed is null for many-to-one (authz
  excluded the child rows, not the parent request)

Note: This test depends on the Cedar policy configuration
used by the test harness. If the existing test Cedar setup
does not support per-table restrictions, this test may need
to be adapted or marked as a known limitation to test
manually.

## Test Setup

- Use `TEST_DATABASE_URL` environment variable (same as existing
  integration tests)
- Skip the entire describe block if `TEST_DATABASE_URL` is not set
- `before`: create pool, run EMBEDDING_SCHEMA_SQL, create pgrest
- `beforeEach`: re-seed data, refresh schema cache
- `after`: drop tables, end pool
- Use `service_role` for all tests unless specifically testing
  authz behavior

## Acceptance Criteria

- All tests compile and run without syntax errors
- All tests FAIL with clear error messages (the feature does
  not exist yet — the parser will reject embed syntax, the
  schema cache has no relationships, etc.)
- No changes to any source files under `src/`
- Existing integration tests still pass (run the full test suite)

## Conflict Criteria

- If any test that is expected to fail instead passes, first
  diagnose why by following the 'Unexpected test results' steps
  in the implementer prompt: investigate the code path, verify
  the assertion targets the right behavior, and attempt to
  rewrite the test to isolate the intended path. Only escalate
  if you cannot construct a well-formed test that targets the
  desired behavior.
- If the embedding test schema conflicts with the existing test
  schema (table name collisions), rename the embedding tables
  to avoid conflicts.

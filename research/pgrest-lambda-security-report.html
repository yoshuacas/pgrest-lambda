1. Executive Summary

This report presents the findings from a comprehensive security audit of pgrest-lambda, a serverless REST API engine that auto-generates PostgREST-compatible endpoints from PostgreSQL schemas, deployed on AWS Lambda.

The audit covered all 25+ source files across REST handling, authentication, authorization (Cedar policies), database adapters, and infrastructure configuration. The analysis applies the STRIDE threat model framework and identifies specific, exploitable vulnerabilities with remediation guidance.

Overall Risk Assessment: MEDIUM-HIGH. The codebase demonstrates strong security fundamentals — parameterized SQL queries, schema-validated identifiers, bulk operation protection, and Cedar-based authorization. However, several issues require attention before production deployment, particularly around JWT secret management, CORS configuration, SSL validation, identifier injection in edge cases, and information disclosure through error messages.

4. Critical & High Severity Findings

CRITICAL V-01: No JWT Secret Strength Enforcement
Affected: src/auth/jwt.mjs:6-8, src/authorizer/index.mjs:8

Description

The JWT secret is consumed directly from config/environment (JWT_SECRET) with no validation of minimum length, entropy, or format. The jsonwebtoken library accepts any string, including empty strings. A weak secret (e.g., "secret", "changeme") allows offline brute-force of all JWTs.

Attack Scenario

1. Attacker obtains any JWT (the apikey is typically shared with all frontend clients). 2. Attacker runs hashcat or jwt-cracker against the token. With a weak secret, this takes seconds. 3. Attacker forges tokens with role: "service_role", gaining full access to all data and bypassing all Cedar policies.

Recommended Fix

// src/auth/jwt.mjs
export function createJwt(config) {
  const secret = config.jwtSecret;
  if (!secret || secret.length < 32) {
    throw new Error(
      'JWT_SECRET must be at least 32 characters. '
      + 'Generate one with: node -e "console.log(require(\'crypto\').randomBytes(32).toString(\'base64\'))"'
    );
  }
  // ... rest unchanged
}
CRITICAL V-02: JWT Algorithm Not Pinned — Algorithm Confusion Risk
Affected: src/auth/jwt.mjs:25-27, src/authorizer/index.mjs:16-18,28-29

Description

The jwt.verify() calls do not specify algorithms: ['HS256']. The jsonwebtoken library defaults to accepting all algorithms including RS256. If an attacker crafts a token using the none algorithm (in older library versions) or RS256 with the HMAC secret as a public key, verification could be bypassed. While jsonwebtoken@9+ blocks the none algorithm, not pinning the algorithm remains a defense-in-depth failure and a risk if the library is ever downgraded.

Attack Scenario

1. Attacker crafts JWT with alg: "HS256" header but uses an empty or guessed secret. 2. In algorithm confusion attacks against RS256/HS256 mismatches, the public key is used as HMAC secret. 3. Even without confusion, failing to pin the algorithm violates JWT best practices (RFC 8725 Section 3.1).

Recommended Fix

// In jwt.mjs:
function verifyToken(token) {
  return jwt.verify(token, secret, {
    issuer: ISSUER,
    algorithms: ['HS256'],
  });
}

// In authorizer/index.mjs (both verify calls):
jwt.verify(apikey, secret, { issuer: ISSUER, algorithms: ['HS256'] });
jwt.verify(token, secret, { issuer: ISSUER, algorithms: ['HS256'] });
HIGH V-03: CORS Wildcard Origin with Credentials
Affected: src/shared/cors.mjs:1-10

Description

CORS is configured with Access-Control-Allow-Origin: * (wildcard). While browsers prevent sending cookies with wildcard origins, the API uses apikey and Authorization headers for auth (both listed in Access-Control-Allow-Headers). Any malicious website can make authenticated cross-origin requests if the user's browser has the API key or access token available (e.g., stored in localStorage as supabase-js does).

Attack Scenario

1. User visits evil-site.com while logged into a pgrest-lambda application. 2. evil-site.com reads the supabase-js session from localStorage (same-origin policy protects this, but if the API key is public/embedded in the frontend, the attacker uses their own). 3. The wildcard CORS allows cross-origin requests from any domain. 4. With the API key (which is public), the attacker can make anon-level requests. With a stolen Bearer token, full authenticated access.

Recommended Fix

// Make CORS configurable with a default allowlist
export function getCorsHeaders(allowedOrigins, requestOrigin) {
  const origin = allowedOrigins === '*' ? '*'
    : allowedOrigins.includes(requestOrigin) ? requestOrigin : null;
  if (!origin) return {};
  return {
    'Access-Control-Allow-Origin': origin,
    'Access-Control-Allow-Credentials': origin !== '*' ? 'true' : undefined,
    'Access-Control-Allow-Headers': '...',
    // ...
  };
}
HIGH V-04: SSL Certificate Validation Disabled
Affected: src/rest/db/postgres.mjs:31, src/rest/db/dsql.mjs:42

Description

Both database providers set ssl: { rejectUnauthorized: false }. This disables TLS certificate validation, making the database connections vulnerable to man-in-the-middle attacks. For DSQL (which is always TLS), this is especially dangerous as the IAM auth token is transmitted over this connection.

Attack Scenario

1. Attacker with network access (e.g., compromised VPC, misconfigured routing) performs ARP spoofing or DNS hijacking. 2. Attacker presents a self-signed certificate for the database endpoint. 3. Because rejectUnauthorized: false, the connection proceeds. 4. Attacker intercepts all SQL queries (including credentials, user data, IAM tokens).

Recommended Fix

// postgres.mjs: Use the system CA bundle (default), or allow custom CA
ssl: config.ssl ? {
  rejectUnauthorized: config.sslRejectUnauthorized !== false,
  ca: config.sslCa || undefined,
} : undefined,

// dsql.mjs: DSQL uses AWS-issued certs, always validate
ssl: { rejectUnauthorized: true },
HIGH V-05: SQL Identifier Injection via on_conflict Parameter
Affected: src/rest/sql-builder.mjs:444-460

Description

The on_conflict query parameter is split on commas and directly interpolated into the SQL ON CONFLICT clause with double-quoting but no schema validation. Unlike column names in SELECT/WHERE (which are validated via validateCol()), the conflict columns are not checked against the schema. An attacker can inject arbitrary identifiers.

While PostgreSQL's double-quoting prevents breaking out of the identifier context (you can't inject full SQL), an attacker can reference columns from other tables or trigger error messages that leak schema information. More critically, if a column name contains a double-quote character that isn't properly escaped, SQL injection becomes possible.

Attack Scenario

POST /rest/v1/items?on_conflict=id"); DROP TABLE items; --
# Results in: ON CONFLICT ("id"); DROP TABLE items; --") DO UPDATE SET ...
# PostgreSQL will reject this syntax, but more subtle injection is possible.

# Identifier with embedded quote:
POST /rest/v1/items?on_conflict=a",(SELECT password FROM auth.users LIMIT 1))--
# Could produce: ON CONFLICT ("a",(SELECT password FROM auth.users LIMIT 1))--")

Recommended Fix

// sql-builder.mjs, buildInsert(), around line 444
if (parsed.onConflict) {
  const conflictCols = parsed.onConflict.split(',').map(c => {
    const col = c.trim();
    validateCol(schema, table, col); // ADD THIS: validates against schema
    return `"${col}"`;
  }).join(', ');
  // ... rest unchanged
}
HIGH V-06: No RLS — Cedar is the Only Authorization Layer
Affected: src/rest/cedar.mjs, src/rest/handler.mjs

Description

The system relies entirely on Cedar policies (evaluated in-process as WASM) for authorization. There is no PostgreSQL Row-Level Security (RLS) as a secondary enforcement layer. If Cedar policies fail to load (empty ./policies directory), the system fails closed (403). However, if there's a bug in the Cedar-to-SQL translation (translateExpr), or if Cedar policies are misconfigured to be overly permissive, there is no database-level safety net.

Additionally, Cedar authorization is only enforced via query filters. The SQL executes as a single database user with full access to the public schema. A SQL injection or Cedar bypass would give access to all data.

Attack Scenario

1. Developer deploys with permit(principal, action, resource); (allow-all) Cedar policy. 2. Anonymous users gain full CRUD access to all tables. 3. Or: a bug in translateExpr() produces a tautological WHERE clause (always true), bypassing row-level filtering.

Recommended Fix

  • Document that Cedar is the primary authorization layer and policies must be audited
  • Add a policy linter/validator that warns about overly permissive policies (e.g., no conditions on permit)
  • Consider adding PostgreSQL RLS as a defense-in-depth layer for sensitive deployments
  • Log all authorization decisions for audit purposes
HIGH V-07: Provider Refresh Token Embedded in JWT (prt claim)
Affected: src/auth/jwt.mjs:17-20, src/auth/handler.mjs:104-107,147-150

Description

The signRefreshToken() function embeds the provider's refresh token (Cognito's refresh token or GoTrue's opaque database token) directly in the JWT's prt claim. JWTs are base64-encoded (not encrypted), so anyone who possesses the refresh token JWT can extract the provider refresh token by decoding the payload.

For Cognito: the extracted Cognito refresh token can be used directly against the Cognito API, bypassing pgrest-lambda's auth layer entirely. For GoTrue: the opaque token is a database lookup key — while less directly exploitable, it's still a credential that shouldn't be in a client-visible token.

Attack Scenario

1. Attacker intercepts or obtains a refresh token JWT (from client storage, logs, or network). 2. Attacker base64-decodes the JWT payload, extracting the prt (Cognito refresh token). 3. Attacker calls Cognito directly with the refresh token, obtaining Cognito access tokens that bypass pgrest-lambda authorization.

Recommended Fix

// Option 1: Encrypt the prt claim (JWE)
// Option 2: Store provider tokens server-side (database or cache)
// and reference them by an opaque session ID in the JWT
function signRefreshToken(sub, sessionId) {
  return jwt.sign(
    { sub, role: 'authenticated', sid: sessionId },
    secret,
    { issuer: ISSUER, expiresIn: '30d' }
  );
}
// Store the provider refresh token in auth.sessions table keyed by sessionId

5. Medium Severity Findings

MEDIUM V-08: Pre-Signup Trigger Auto-Confirms All Users
Affected: src/presignup.mjs:1-5

Description

The Cognito pre-signup trigger automatically confirms all users and verifies their email without any actual email verification. This means anyone can sign up with any email address (including emails they don't own), and they'll immediately have a confirmed account.

Attack Scenario

Attacker signs up as ceo@company.com and gets an immediately confirmed account. If any Cedar policies use email-based access control, the attacker gains those permissions.

Recommended Fix

Remove auto-confirmation and implement proper email verification flow, or clearly document this as a development-only convenience and ensure it's disabled in production.

MEDIUM V-09: Error Messages Leak Internal Database Details
Affected: src/rest/errors.mjs:48-57, src/rest/handler.mjs:340-354

Description

PostgreSQL errors are forwarded to the client with their original message, detail, and hint fields. These can contain table names, column names, constraint names, SQL fragments, and internal paths. The catch-all error handler also forwards err.message for unrecognized errors.

Attack Scenario

// Attacker sends a malformed insert to discover constraint names:
POST /rest/v1/users {"email": "dup@test.com", "name": "x"}
// Response: { "code": "23505", "message": "duplicate key value violates
//   unique constraint \"users_email_key\"", "detail": "Key (email)=(dup@test.com)
//   already exists." }
// This reveals: constraint names, column names, and actual values

Recommended Fix

// Sanitize PG error details for client responses
export function mapPgError(pgError) {
  const statusCode = PG_ERROR_MAP[pgError.code] || 500;
  const safeMessages = {
    '23505': 'Resource already exists',
    '23503': 'Referenced resource not found',
    '23502': 'Required field missing',
  };
  return new PostgRESTError(
    statusCode,
    pgError.code,
    safeMessages[pgError.code] || 'Database error',
    null,  // Never forward pgError.detail to client
    null,
  );
}
MEDIUM V-10: OpenAPI Spec Exposes Full Schema to Unauthenticated Users
Affected: src/rest/handler.mjs:133-137, src/rest/openapi.mjs

Description

The OpenAPI spec at GET /rest/v1/ includes all table names, column names, data types, nullability, and required fields for every table in the public schema. While the Lambda authorizer protects REST endpoints, the OpenAPI endpoint itself performs no role-based filtering. If the authorizer allows anon access (which it does by design for the API key), the full database structure is visible.

Recommended Fix

Consider optionally restricting the OpenAPI spec to authenticated users, or filtering it based on the requester's role and Cedar policies.

MEDIUM V-11: Schema Refresh Endpoint Has No Authorization Check
Affected: src/rest/handler.mjs:151-159

Description

The POST /rest/v1/_refresh endpoint refreshes the schema cache and Cedar policies without checking the caller's role. Any authenticated user (including anon-key holders) can trigger a schema refresh. This forces database introspection queries and S3 reads (for Cedar policies), creating a potential DoS vector. It also refreshes Cedar policies, meaning if S3 policies were temporarily modified, a timed _refresh call could load malicious policies.

Recommended Fix

if (routeInfo.type === 'refresh') {
  if (method !== 'POST') { /* ... */ }
  if (role !== 'service_role') {
    throw new PostgRESTError(403, 'PGRST403', 'Only service_role can refresh schema');
  }
  // ...
}
MEDIUM V-12: No Rate Limiting on Auth Endpoints
Affected: src/auth/handler.mjs

Description

Auth endpoints (signup, token/password, token/refresh) have no rate limiting. While API Gateway can be configured with usage plans, the application layer has no protection. The bcrypt comparison in GoTrue provides some protection against rapid brute-force (bcrypt cost=10 takes ~100ms), but this doesn't prevent credential stuffing or account enumeration attacks.

Attack Scenario

Attacker sends thousands of login attempts against /auth/v1/token?grant_type=password with common passwords. The timing-safe bcrypt comparison prevents timing attacks, but the volume can still succeed against weak passwords.

Recommended Fix

Implement rate limiting at the API Gateway level (usage plans/throttling) and add per-IP or per-email lockout after N failed attempts at the application layer.

MEDIUM V-13: Unbounded Resource Embedding Depth
Affected: src/rest/sql-builder.mjs:115-174, src/rest/query-parser.mjs:19-90

Description

While the query parser limits logical operator nesting to 10 levels, resource embedding (joins) has no depth limit. An attacker can craft deeply nested embeds like select=*,a(b(c(d(e(f(g(h(i(j(*))))))))))) which generates deeply nested correlated subqueries. Each level adds a subquery, and PostgreSQL's query planner may take exponential time.

Recommended Fix

// In parseSelectList, add depth tracking:
export function parseSelectList(input, depth = 0) {
  if (depth > 5) {
    throw new PostgRESTError(400, 'PGRST100',
      'Resource embedding depth exceeds maximum of 5');
  }
  // ... when recursing for embeds:
  const childNodes = parseSelectList(innerContent, depth + 1);
}
MEDIUM V-14: Order Direction Not Validated — SQL Injection via ORDER BY
Affected: src/rest/query-parser.mjs:328-337, src/rest/sql-builder.mjs:290-301

Description

The parseOrder() function splits order entries by dots but does not validate the direction value. The direction is then interpolated into SQL via .toUpperCase() without validation. An attacker could inject arbitrary SQL after the ORDER BY clause.

// query-parser.mjs:330-335
const parts = entry.split('.');
return {
  column: parts[0],
  direction: parts[1] || 'asc',  // NOT VALIDATED
  nulls: parts[2] || null,       // NOT VALIDATED
};

// sql-builder.mjs:294
let sql = `"${o.column}" ${o.direction.toUpperCase()}`;
// If direction is "asc; DROP TABLE users--", this becomes:
// "col" ASC; DROP TABLE USERS--

Recommended Fix

const VALID_DIRECTIONS = new Set(['asc', 'desc']);
const VALID_NULLS = new Set(['nullsfirst', 'nullslast']);

function parseOrder(raw) {
  return raw.split(',').map((entry) => {
    const parts = entry.split('.');
    const direction = (parts[1] || 'asc').toLowerCase();
    if (!VALID_DIRECTIONS.has(direction)) {
      throw new PostgRESTError(400, 'PGRST100',
        `Invalid order direction: ${direction}`);
    }
    const nulls = parts[2] ? parts[2].toLowerCase() : null;
    if (nulls && !VALID_NULLS.has(nulls)) {
      throw new PostgRESTError(400, 'PGRST100',
        `Invalid nulls option: ${nulls}`);
    }
    return { column: parts[0], direction, nulls };
  });
}

6. Low & Informational Findings

LOW V-15: Schema Cache Race Condition
Affected: src/rest/schema-cache.mjs:175-183

The schema cache getSchema() function has a TOCTOU race: multiple concurrent requests can trigger simultaneous introspection queries if the TTL expires at the same time. In Lambda's single-concurrency model this is unlikely, but in high-concurrency deployments, it wastes database resources.

Fix: Add a mutex or "refreshing" flag to coalesce concurrent refreshes.

LOW V-16: Cedar Policy Fail-Closed but No Observability
Affected: src/rest/cedar.mjs:349-395

When Cedar policies cannot be loaded or evaluation fails, the system correctly returns 403 (fail-closed). However, there is no logging of policy evaluation results, denials, or errors. This makes it impossible to debug authorization issues or detect attacks in production.

Fix: Add structured logging for Cedar authorization decisions (allow/deny, policy ID, principal, resource, residual conditions).

LOW V-17: Cognito ID Token Parsed Without Signature Verification
Affected: src/auth/providers/cognito.mjs:24-45

The parseIdToken() function decodes the Cognito ID token's payload via base64 without verifying the JWT signature. Since this token comes from the Cognito SDK response (over TLS from AWS), the risk is limited. However, if the response were tampered with (e.g., via a man-in-the-middle, especially since V-04 disables cert validation), the user identity could be spoofed.

Fix: Verify the ID token signature using Cognito's JWKS endpoint, or at minimum validate the token structure and expected claims.

LOW V-18: JSON Body Parse Failure Silently Sets body=null
Affected: src/rest/handler.mjs:114-119

When JSON.parse(event.body) throws, the catch block silently sets body = null. For POST (insert), this triggers the "Missing or invalid request body" error. But for PATCH, the check is !body || typeof body !== 'object' which also catches null. The silent swallowing of the parse error means the client doesn't know their JSON was malformed — they just see a generic error.

Fix: Return a specific 400 error: "Request body is not valid JSON".

LOW V-19: No Request Size Limits
Affected: src/rest/handler.mjs (system-wide)

There are no checks on request body size. API Gateway has a 10MB limit by default, but bulk inserts with thousands of rows generate proportionally large SQL queries and parameter arrays, consuming Lambda memory.

Fix: Add a configurable row limit for bulk inserts (e.g., max 1000 rows per request).

INFO V-20: XSS Vector in docsHtml via Host Header
Affected: src/rest/handler.mjs:47-59,91-94

The docsHtml() function injects specUrl into an HTML string via template literal: data-url="${specUrl}". The specUrl is derived from the Host header. If an attacker can control the Host header, they could inject HTML/JavaScript. API Gateway typically normalizes the Host header, mitigating this in production. However, in local development or behind certain proxies, this could be exploitable.

Fix: HTML-escape the specUrl value before interpolation.

INFO V-21: Refresh Token Rotation Race Condition Window
Affected: src/auth/providers/gotrue.mjs:101-146

The refresh token rotation in GoTrue has a small race window: between the SELECT (line 105-109) and UPDATE (line 136-140), a concurrent request could also read the same non-revoked token. The code handles token reuse detection (revoked=true triggers family revocation), which is good. However, the window between read and revoke is not atomic (no SELECT ... FOR UPDATE), meaning two concurrent refreshes could both succeed.

Fix: Use SELECT ... FOR UPDATE or UPDATE ... RETURNING for atomic token consumption.

INFO V-22: Auth Schema Auto-Migration in Production
Affected: src/auth/schema.mjs:34-46

The ensureAuthSchema() function runs CREATE SCHEMA IF NOT EXISTS auth and CREATE TABLE IF NOT EXISTS on every cold start. While idempotent, running DDL from the application runtime is a code smell. The initialized flag is module-level, so it resets on Lambda cold starts. If the database user lacks CREATE permissions, every auth request will fail with an unhelpful error.

Fix: Consider moving schema migration to a separate setup step/CLI command, and have the runtime only validate the schema exists.

INFO V-23: No Audit Logging
Affected: System-wide

There is no structured logging for security-relevant events: authentication attempts (success/failure), authorization decisions, data mutations, schema refreshes, or error conditions. This creates a repudiation risk and makes incident investigation impossible.

Fix: Add structured JSON logging for: auth events (signup, login success/failure, token refresh, logout), authorization decisions (Cedar allow/deny with principal and resource), mutations (INSERT/UPDATE/DELETE with table name and row count), and errors.

9. Positive Security Observations

The codebase demonstrates several strong security practices that should be recognized and maintained:

  1. Consistent SQL parameterization — Every user-supplied value flows through $N placeholders. No string concatenation of values into SQL. This is the single most important security property of the codebase.
  2. Schema-validated identifiers — Column names are checked against the introspected schema before being included in SQL. This prevents most identifier injection attacks.
  3. Bulk operation protection — UPDATE and DELETE require at least one filter, preventing accidental mass data modification.
  4. Cedar fail-closed design — When policies cannot be loaded or evaluation fails, the system denies access (403) rather than failing open.
  5. Timing-safe auth — The GoTrue provider uses a dummy bcrypt comparison for non-existent users, preventing user enumeration via timing differences.
  6. Refresh token rotation — Token reuse detection with family revocation is a strong defense against stolen refresh tokens.
  7. Defense-in-depth auth model — The dual-layer auth (apikey + Bearer token) means even if one layer is compromised, the attack surface is limited.
  8. Input validation at parser level — Operators are validated against an allowlist, IS values are restricted, and nesting depth is capped.
  9. Clean error abstraction — The PostgRESTError class provides structured errors without raw exception forwarding (though PG error mapping needs work per V-09).
  10. No-store cache control — CORS headers include Cache-Control: no-store, preventing proxy caching of API responses.
